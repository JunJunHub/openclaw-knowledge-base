Superpowers 是一个 **AI 编码助手技能框架**，为 Claude Code、Cursor、Codex 等 AI 编程工具提供完整的软件开发工作流，将 Claude Code 从"有时偏离轨道的助手"转变为"遵循成熟流程的系统化开发伙伴"。

---

## 🚀 核心功能总览

### 1. 结构化工作流（预防常见失败模式）
- **自动拦截**：在关键节点拦截 Claude Code，防止直接跳入编码
- **Socratic 提问**：通过提问澄清需求，探索替代方案
- **分阶段执行**：强制遵循"设计→计划→执行→验证"流程

### 2. 测试驱动开发 (TDD) 强制执行
- **RED-GREEN-REFACTOR** 循环：先编写失败测试 → 验证失败 → 实现代码 → 验证通过 → 重构
- **代码删除机制**：如果 Claude 在测试前编写代码，技能会指示删除并重新开始

### 3. 子代理驱动开发
- **每个任务独立子代理**：为每个任务生成全新上下文，避免长时间会话积压混淆
- **两阶段审查**：规格合规性 + 代码质量双重检查
- **并发子代理工作流**：支持并行任务执行

### 4. Git 工作流集成
- **Git Worktree 隔离**：为每个功能创建独立分支和工作区
- **开发分支管理**：提供合并/PR/保留/丢弃选项
- **主分支保护**：保持主分支稳定，避免实验性代码污染

## 🎯 三个核心命令详解

### 1. `/superpowers:brainstorm` - 设计前头脑风暴
**用途**：在任何新功能、重构或迁移之前使用，通过提问澄清目标、探索替代方案。

**执行过程**：
1. **提问澄清**：询问关于目标的详细信息
2. **探索替代**：提供你可能未考虑的其他方案
3. **分块呈现**：以可消化的块展示设计供验证
4. **设计文档**：生成参考设计文档

**何时触发**：
- 新功能开始时
- 代码重构前
- 技术迁移前
- 重大修改决策前

### 2. `/superpowers:write-plan` - 创建详细实施计划
**用途**：在头脑风暴后，创建分步实施计划。

**计划特点**：
- **任务粒度**：每个任务 2-5 分钟完成
- **精确路径**：每个任务包含确切文件路径和完整代码片段
- **验证步骤**：每个任务内置验证步骤
- **上下文独立**：计划假设"没有上下文的热心初级工程师"能执行

### 3. `/superpowers:execute-plan` - 执行计划
**用途**：使用子代理驱动开发执行计划。

**执行流程**：
1. **任务分派**：每个任务生成独立的子代理
2. **两阶段审查**：
   - 第一阶段：规范合规性审查
   - 第二阶段：代码质量审查
3. **检查点**：批次间设置人工检查点
4. **阻塞机制**：关键问题自动阻止进度

| **系统化调试** | `systematic-debugging` | 4 阶段根本原因分析，深度防御 | 遇到 bug 时 |
| **使用 Git Worktree** | `using-git-worktrees` | 创建隔离分支，运行设置，验证干净基线 | 开发开始时 |
| **使用 Superpowers** | `using-superpowers` | 技能系统介绍 | 首次使用时 |
| **派发并行代理** | `dispatching-parallel-agents` | 并发子代理工作流 | 需要并行执行时 |
| **执行计划** | `executing-plans` | 带检查点的批量执行 | 计划准备就绪时 |
| **完成开发分支** | `finishing-a-development-branch` | 合并/PR/保留/丢弃选项，清理 | 所有任务完成后 |
| **头脑风暴** | `brainstorming` | Socratic 设计细化 | 讨论新功能时 |
| **编写计划** | `writing-plans` | 详细的实施计划 | 设计批准后 |
| **请求代码审查** | `requesting-code-review` | 预审查清单，关键问题阻塞 | 任务完成后 |
| **接收代码审查** | `receiving-code-review` | 系统化响应反馈 | 收到审查意见时 |
| **编写技能** | `writing-skills` | 创建新技能的元技能 | 扩展功能时 |
| **完成前验证** | `verification-before-completion` | 确保修复实际有效 | 修复完成后 |
| **子代理驱动开发** | `subagent-driven-development` | 每个任务两阶段审查 | 执行多任务时 |

### 技能自动触发机制
技能不是手动调用的命令，而是**基于上下文自动激活**：

1. **开始讨论新功能** → **头脑风暴**激活
2. **设计批准** → **使用 Git Worktree**激活  
3. **实施开始** → **测试驱动开发**激活
4. **任务完成** → **请求代码审查**激活
5. **所有任务完成** → **完成开发分支**激活

Claude 在任何任务前都会检查相关技能，这些是**强制工作流**，不是建议。

## 🔄 完整开发流程示例

### 场景：为 Express 应用添加用户认证

#### 第 1 步：设计头脑风暴
```
用户：我需要为我的 Express 应用添加用户认证
Claude（激活大脑风暴技能）：
→ 提问：OAuth 还是密码？Session 还是 JWT？支持哪些提供者？
→ 探索：速率限制？账户锁定？密码要求？
→ 分块呈现设计供您批准
```

#### 第 2 步：创建详细计划
```
用户：设计看起来不错，开始计划
Claude（激活编写计划技能）：
→ 创建 12 个任务，每个 2-5 分钟
→ 任务 1：为 /signup 端点创建失败测试
→ 任务 2：实现 signup 处理器通过测试
→ 每个任务包含确切文件路径和代码
```

#### 第 3 步：执行计划
```
用户：执行
Claude（激活子代理驱动开发）：
→ 为任务 1 生成子代理
→ code-reviewer 代理验证输出
→ 为任务 2 生成子代理  
→ 报告进度，请求继续批准
```

#### 第 4 步：完成与验证
```
Claude（激活完成开发分支）：
→ 验证所有测试通过
→ 提供选项：1) 创建 PR 2) 直接合并 3) 保留分支 4) 丢弃
→ 清理临时文件
```

## 💻 Git Worktree 工作机制

### 后台自动执行
```bash
# Superpowers 自动执行的操作：
git worktree add ../project-feature feature-branch
cd ../project-feature
# Claude 在此工作，主分支保持干净
```

### 核心优势
1. **并行开发**：多个 Claude 实例处理不同功能
2. **主分支保护**：实验期间主分支保持稳定
3. **简易清理**：出现问题只需删除工作树
4. **上下文隔离**：每个功能独立设置

## 🏗️ 子代理驱动开发模式

### 与传统单一会话对比
| 模式 | 单一会话 | 子代理驱动 |
|------|----------|------------|
| **上下文** | 会话越长，越混乱 | 每个任务全新上下文 |
| **专注度** | 多任务交叉，易分心 | 一次只专注一个任务 |
| **错误传播** | 早期错误影响后期 | 任务间隔离 |
| **审查质量** | 自我审查，缺乏客观性 | 独立审查代理 |

### 工作流程
1. **主代理创建计划**：分析需求，分解任务
2. **子代理执行任务**：每个任务使用独立实例
3. **审查代理检查**：`code-reviewer` 代理双重检查
4. **主代理协调**：处理进度、请求审批、继续或修复

## ✅ TDD 强制执行机制

### 严格的 TDD 循环
```
1. RED：编写会失败的测试 - 验证测试实际测试了某些内容
2. WATCH：观看测试失败 - 证明它确实在测试
3. GREEN：编写最少代码通过测试
4. COMMIT：在绿色时提交
5. REFACTOR：仅在绿色后重构
```

### 违规处理
如果 Claude 在测试前编写代码：
- 技能指示删除该代码
- 重新开始正确的 TDD 流程
- 解释为何测试优先至关重要

## 🎯 何时使用（与不使用）Superpowers

### ✅ 推荐使用场景
- **多文件功能**：涉及多个文件修改的新功能
- **代码重构**：大规模代码结构调整
- **技术迁移**：框架/库版本更新
- **复杂调试**：需要系统化分析的 bug
- **团队协作**：需要标准化流程的团队项目

### ❌ 不适合场景  
- **一次性问题**：快速的一次性查询
- **简单编辑**：单个文件的微小修改
- **5 分钟任务**：预计完成时间 <5 分钟的任务
- **探索性编码**：不确定目标的实验性代码

在这些情况下，开销不值得，直接使用 Claude Code 更高效。

## 🚀 最佳实践指南

### 1. 从头脑风暴开始
永远从 `/superpowers:brainstorm` 开始任何非平凡任务。设计阶段节省的时间比执行阶段多。

### 2. 信任计划过程
让 Claude 创建详细计划，即使你觉得可以直接编码。计划确保没有遗漏的细节。

### 3. 使用 Git Worktree
让 Superpowers 管理 Git 工作树，保持主分支干净，便于并行工作。

### 4. 利用子代理隔离
对于复杂任务，主动建议使用子代理驱动开发，获得更好的上下文隔离。

### 5. 尊重检查点
当 Claude 请求继续批准时，花时间真正审查进度，不要盲目点击"继续"。

### 6. 配合其他工具
- **结合 SpecSwarm**：用于一体化生命周期管理
- **配合 GitHub Spec-Kit**：用于企业级标准化
- **使用 Cursor 技能**：在不同 IDE 间保持一致流程

## 🐛 常见问题与解决方案

### Q1：Superpowers 使 Claude 变慢了吗？
**A**：是的，但这是有意设计的。额外的结构防止了后续的重做和调试，从长远来看节省了时间。

### Q2：可以跳过某些步骤吗？
**A**：如果你确定，可以通过明确指示跳过某些阶段，但这违反了工具的设计理念。

### Q3：如何处理现有大型代码库？
**A**：使用 `using-git-worktrees` 创建隔离分支。对于 Brownfield 项目，考虑结合 GitHub Spec-Kit 的 Brownfield 扩展。

### Q4：Superpowers 与 GitHub Spec-Kit 有什么区别？
**A**：Superpowers 是更轻量、更专注于 Claude Code 的技能框架，而 Spec-Kit 是企业级、多工具的工作流框架。

### Q5：我可以扩展技能吗？
**A**：是的！使用 `writing-skills` 元技能来创建自定义技能，满足特定需求。

## 🎓 总结：为什么使用 Superpowers？

Superpowers 不是锦上添花，而是**AI 辅助编程的必要基础架构**。它解决了 Claude Code 的核心问题：

1. **预防跳入编码**：通过结构化提问确保理解需求
2. **强制执行最佳实践**：TDD、代码审查等不再是可选项
3. **管理复杂性**：通过子代理隔离和 Git 工作树保持秩序
4. **确保质量**：多阶段验证防止回归错误

对于严肃的软件开发，Superpowers 提供的结构化和系统化方法，是将 AI 从"有用的助手"提升为"可靠开发伙伴"的关键转变。
